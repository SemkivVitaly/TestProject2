# Подключение к базе данных по локальной сети (SaveData1)

Инструкция для администратора: один компьютер с **Microsoft SQL Server** и базой **SaveData**, остальные ПК — клиенты с приложением **SaveData1**.

---

## 1. Общая схема

- Клиенты подключаются к SQL Server по **имени компьютера или IP-адресу** и **TCP-порту** (часто **1433** для экземпляра по умолчанию).
- По умолчанию SQL Server может не принимать внешние подключения, а брандмауэр Windows — блокировать порт.
- В приложении строка подключения собирается из пользовательских настроек (**сервер**, **режим проверки подлинности**, логин/пароль при SQL-аутентификации) — см. раздел 4.

---

## 2. Настройка SQL Server на компьютере с базой

### 2.1. Режим проверки подлинности

Для сети из ПК в **рабочей группе** (без домена Active Directory) обычно проще включить **смешанный режим** и создать **отдельный логин SQL Server**:

1. Откройте **SQL Server Management Studio (SSMS)** и подключитесь к нужному экземпляру.
2. Правый клик по серверу → **Свойства** → раздел **Безопасность (Security)**.
3. Включите **SQL Server and Windows Authentication mode** (смешанный режим).
4. Подтвердите изменения и **перезапустите** службу **SQL Server** (через «Службы» Windows или Configuration Manager).

*Альтернатива:* только **Windows Authentication** — удобнее в **домене**, когда для группы пользователей AD создают логин в SQL Server. В простой рабочей группе чаще используют **SQL login** (раздел 3).

### 2.2. Протокол TCP/IP

1. Запустите **SQL Server Configuration Manager** (диспетчер конфигурации SQL Server).
2. **Конфигурация сети SQL Server** → **Протоколы для &lt;имя_экземпляра&gt;**.
3. Включите протокол **TCP/IP** (Enabled).
4. Откройте свойства **TCP/IP** → вкладка **IP Addresses** → в секции **IPAll** задайте **TCP Port** (для экземпляра по умолчанию часто указывают **1433**). Для **именованного** экземпляра по умолчанию может быть динамический порт — тогда либо зафиксируйте порт здесь, либо используйте **SQL Server Browser** (см. ниже).

**Перезапустите** службу **SQL Server**.

### 2.3. Именованный экземпляр и SQL Server Browser

Если экземпляр не «по умолчанию» (например, `ИМЯ_ПК\SQLEXPRESS`), клиенту может понадобиться:

- строка вида **`ИМЯ_ПК\ИМЯ_ЭКЗЕМПЛЯРА`**, или  
- **`tcp:ИМЯ_ПК,ПОРТ`** при **фиксированном** TCP-порту.

Службу **SQL Server Browser** нужно запустить и разрешить в брандмауэре (UDP **1434**), если полагаетесь на автоматическое определение порта именованным экземпляром.

### 2.4. Брандмауэр Windows на сервере

Создайте **входящее** правило:

- **Порт** → **TCP** → указите порт, на котором слушает SQL Server (например, **1433**),  
  **или**  
- разрешите программу **`sqlservr.exe`** (каталог `Binn` установки SQL Server).

Без этого клиенты в локальной сети не достучатся до сервера.

### 2.5. Проверка доступности порта с клиента

На любом ПК в сети (PowerShell):

```powershell
Test-NetConnection -ComputerName IP_ИЛИ_ИМЯ_СЕРВЕРА -Port 1433
```

Для успешного подключения к SQL по TCP в поле **TcpTestSucceeded** должно быть **True** (при использовании именно этого порта).

## 3. Учётная запись SQL для приложения

В **SSMS** на сервере:

1. **Безопасность** → **Имена входа** → **Создать** → тип **Проверка подлинности SQL Server**.
2. Задайте **имя входа** и **надёжный пароль**.
3. Вкладка **Сопоставление пользователей** → отметьте базу **SaveData** → включите роли, как минимум:
   - **db_datareader**
   - **db_datawriter**

При необходимости (если приложение выполняет особые операции со схемой) добавьте другие роли по согласованию с разработчиком.

Сохраните **логин и пароль** — они понадобятся при настройке каждого клиента.

---

## 4. Настройка приложения SaveData1 на клиентских ПК

Подключение формируется в коде из пользовательских параметров: **имя сервера**, флаг **использовать SQL-аутентификацию**, **логин** и **пароль**.

На **каждом** компьютере, где запускают программу:

1. Установите приложение (exe и зависимости, как принято у вас).
2. Откройте в программе окно **настроек подключения к БД** (форма настроек сервера и авторизации).
3. Укажите:
   - **сервер**: IP-адрес или имя ПК с SQL; для именованного экземпляра — `ИМЯ\ЭКЗЕМПЛЯР` или `tcp:ИМЯ,ПОРТ`;
   - включите **аутентификацию SQL Server**;
   - введите **логин** и **пароль**, созданные в разделе 3;
4. Сохраните настройки и проверьте вход в приложение.

Настройки хранятся **для пользователя Windows** на этом ПК: при первом запуске под другим пользователем Windows на том же компьютере поля нужно заполнить снова.

---

## 5. Рекомендации по сети

- Задайте на сервере **статический IP** в подсети или **резервацию DHCP** по MAC-адресу, чтобы адрес не менялся.
- Убедитесь, что клиенты **пингуются** с сервера и наоборот по IP (базовая проверка маршрутизации).

---

## 6. Вариант только Windows Authentication

Если все ПК в **домене Active Directory**, можно не создавать SQL-логин, а выдавать права **группе домена** или пользователям через логин Windows в SQL Server. Тогда в настройках приложения **не** включают SQL-аутентификацию (используется встроенная проверка Windows текущего пользователя). Для рабочей группы без домена этот путь обычно сложнее, чем смешанный режим и SQL login.

---

## 7. Краткий чеклист

| № | Действие |
|---|----------|
| 1 | Включить смешанный режим проверки подлинности в SQL Server |
| 2 | Включить TCP/IP, задать/знать порт, перезапустить SQL Server |
| 3 | Открыть порт (или sqlservr.exe) в брандмауэре сервера |
| 4 | Создать SQL login и пользователя в базе SaveData с reader/writer |
| 5 | На каждом клиенте в настройках приложения указать сервер и SQL login |
